# Phase 2 — Structured extraction (10 components)

**Purpose:** Extract and organize content into the 10 messaging components. This is the core analytical phase. Use the framework definitions in `references/frameworks.md` for the descriptor counts and link rules; use the templates in `references/output-template.md` for the exact output shape.

---

## Steps

### Step 2.1: Description (Component 1)

- Extract core positioning statement
- Include: what it is, who it serves, primary value, key differentiator
- **Output:** 1 rich paragraph with source

If JTBD framing adds clarity, include as Component 0 per `references/frameworks.md` ("JTBD statement").

### Step 2.2: Status quo and alternatives (Component 2)

- Document Manual / DIY approaches
- Document Legacy / incumbent tools
- Document Direct competitors
- Document Adjacent alternatives
- Include market category for each
- **Output:** 3-5 alternatives with pain point mappings

Status quo is often the real competitor (doing nothing, spreadsheets, "we'll just hire someone"). Don't skip Manual/DIY.

### Step 2.3: Pain points with consequence chains (Component 3)

- For each pain point, trace:
  - 1st order (direct): Immediate tangible problem
  - 2nd order (operational): Downstream impact
  - 3rd order (strategic): Business-level consequence
- Link backward to status quo, forward to capability
- **Output:** 3-5 pain points with full consequence chains

Consequence chain framework + worked example → `references/frameworks.md` ("Consequence Chain Framework").

### Step 2.4: Capabilities (Component 4)

- Document what users can DO
- Link each capability to branded feature name
- Link each capability to pain point it solves
- **Output:** 3-5 capabilities with feature links

### Step 2.5: Functional benefits (Component 5)

- Document outcomes customers achieve
- Include customer quote as proof for each
- Mark "Not available" if no quote found
- **Output:** 3-5 benefits with customer proof

### Step 2.6: Emotional & social benefits (Component 5.5)

- Extract 2 emotional benefits (how users feel)
- Extract 2 social benefits (how others perceive them)
- **Output:** 4 benefits with triggers/contexts

### Step 2.7: Features (Component 6)

- Use EXACT branded feature names (from Phase 1, Step 1.3)
- Document what each does and its benefit
- Link to capabilities each powers
- **Output:** 3-5 features with capability links

### Step 2.8: Cost of inaction (Component 7)

- Quantify ongoing costs of not solving
- Include daily / weekly / monthly cost, cumulative impact, competitive risk
- **Output:** 2-3 cost of inaction statements

### Step 2.9: Common objections (Component 8)

- Extract from FAQ, comparison pages, reviews
- Document root cause and response framework for each
- **Output:** 3-5 objections with responses

### Step 2.10: Core messaging blocks (Component 9)

- Distill tagline (max 7 words)
- Create elevator pitch (1 sentence)
- Create value proposition (3 bullets)
- Extract proof point
- Segment by audience
- **Output:** Reusable messaging blocks

Tagline rule: complete all other components first. Don't open with the tagline — it's a distillation, not a starting point.

---

## Phase 2 checkpoint

Before moving to Phase 3:

- [ ] All 10 components have content or explicit "Not available"
- [ ] All capabilities link to features
- [ ] All pain points have consequence chains
- [ ] All benefits have customer quotes or "Not available"
- [ ] Branded feature names from Phase 1 used verbatim — no paraphrasing
- [ ] Component link graph (per `references/frameworks.md`) traversable end-to-end

---

## Anti-patterns

- ❌ Skipping Component 5.5 (emotional + social benefits) because it feels "soft" — it's load-bearing for content downstream.
- ❌ Generic objection responses ("we have great support") — must trace to root cause.
- ❌ Cost of inaction without quantification — "you'll waste time" is not a cost; "30 min/candidate × 100 candidates/month = 50 hours/month" is.
- ❌ Tagline written before Components 1-8 finalized.

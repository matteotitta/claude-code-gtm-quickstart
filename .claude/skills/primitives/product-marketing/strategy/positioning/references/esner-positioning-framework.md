# Esner 3-Step Positioning Framework — Reference

**Source:** Alexander Esner, 3-Step Positioning Framework
**Integrated into:** Positioning skill v2.4 (Frameworks 13, 14, 15)
**Relationship to Fletch:** Complementary. Fletch provides the anchor taxonomy and differentiation evidence types. Esner provides selection mechanisms, angle archetypes, and proof mapping on top of that taxonomy.

---

## Core concept: 3-step positioning sequence

Esner's framework operates as a three-step sequence that builds on Fletch's anchor system:

1. **Select the primary anchor** — Use the decision tree to determine which anchor type fits your market maturity
2. **Choose a secondary angle** — Pick an archetype that sharpens differentiation within your anchor
3. **Map proof to anchor** — Match evidence strategy to your anchor type for credible positioning

---

## Step 1: Anchor Selection Decision Tree

A top-down sequential test. Run each step in order — the first YES determines your starting hypothesis.

**Step 0: Leader Dominance Test**
- Is there a single dominant leader owning 40%+ of buyer mindshare?
- Do you have a clear wedge against that leader?
- IF YES to both → **Competitive Alternative** anchor

**Step 1: Category Awareness Test**
- Do prospects use a known category name when describing what they need?
- Do they Google a specific software category, compare vendors on G2/Capterra?
- IF YES → **Product Category** anchor

**Step 2: Problem Articulation Test**
- Can prospects describe the specific friction/pain they experience?
- Do they have language for the problem, just not for the solution category?
- IF YES → **Use Case** anchor

**Step 3: Manual Behavior Test**
- Are prospects solving this with manual work (spreadsheets, email, meetings)?
- Is there no established software category for this task?
- IF YES → **Activity** anchor

### Market maturity alignment

| Anchor | Market Maturity | Buyer Awareness | Typical Alternative |
|--------|-----------------|-----------------|---------------------|
| Activity | Pre-market | Low — no category language | Manual/DIY processes |
| Use Case | Emerging | Medium — problem language, no category | Broken workflows, workarounds |
| Product Category | Established | High — category language exists | Direct competitors |
| Competitive Alternative | Mature | Very high — dominant leader exists | Named market leader |

**Important:** This tree produces a hypothesis, not a final answer. Validate with anchor scoring (Step 2.4 in the positioning process).

## Step 2: Secondary Angle Archetypes

After selecting a primary anchor, choose a secondary angle that sharpens differentiation. Maximum of 2.

| Angle Archetype | Formula | When It Works |
|-----------------|---------|---------------|
| **Niche Down** | "[Primary anchor] built for [specific segment]" | ICP is sharply defined, generalists fail them |
| **Low-Cost Player** | "[Primary anchor] without [expensive pain]" | Incumbents are overpriced for segment |
| **Premium Player** | "The most advanced [primary anchor]" | Buyers will pay for best-in-class |
| **Unique Attribute** | "Only [primary anchor] with [exclusive capability]" | You have a genuinely exclusive feature |
| **Lite Version** | "A lightweight way to [primary anchor activity]" | Existing tools are bloated for use case |

### Relationship to differentiation evidence (Framework 9)

- Angle archetypes select the POSITIONING PLAY (what type of differentiation story)
- Differentiation test (Framework 9) validates the EVIDENCE (is it defensible?)
- Example flow: Choose "Unique Attribute" angle → test whether that attribute passes the guarantee test and has proprietary backing

## Step 3: Proof Type Mapping per Anchor

Each primary anchor type demands a different evidence strategy.

| Primary Anchor | Best Proof Type | What It Demonstrates |
|----------------|-----------------|---------------------|
| **Activity** | Before → After transformation | Life before software vs. after |
| **Use Case** | Workflow power + Outcome proof | End-to-end capability AND measurable result |
| **Product Category** | Comparative proof | Head-to-head advantage over category peers |
| **Competitive Alternative** | Switch & Upgrade proof | Migration story + what they gained |

### Downstream activation

- Proof types inform the guarantee test (Step 3.7) — match evidence format to anchor type
- Proof types feed directly into `sales-enablement` battlecards and `product-messaging` proof points
- Case study strategy should prioritize collecting the proof type that matches your primary anchor

---

## Activity vs Use Case distinction

This is the most common source of confusion when selecting anchors.

**Activity positioning** names the raw task without specifying friction:
- "Record yourself" (Loom)
- Works when no software category exists yet and buyers don't have language for the problem

**Use Case positioning** names the task AND the problem:
- "Schedule meetings over email" (Calendly) — the problem is the back-and-forth
- Works when buyers recognize the friction but haven't found a dedicated solution

**Diagnostic:** If buyers say "I need a way to [do X]" → Activity. If buyers say "I'm tired of [problem when doing X]" → Use Case.
